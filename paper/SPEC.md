# The Telephone Game, project specification

Version 0.1, 2026-07-20. This document is the contract for the whole project. Every later artifact (trace, figures, paper) must conform to it or this file must be updated first.

## 1. The question

When a human hands a task to an orchestrating AI, and that AI hands pieces of it to coding subagents, how much of the original intent survives each hop, where exactly does it drift, and how much of the drift is recovered by the review loop?

## 2. What gets built and by whom

The experiment instruments one real build task.

Chosen build task, the **shift report command** for Asish's existing shift tracker. It was chosen because it has a written, testable specification, real value (it ships either way), and enough surface area (data reading, aggregation, formatting, edge cases) for interpretation differences to show up.

Roles.

- Layer 0, Asish. States the intent in his own words, once, without coaching.
- Layer 1, Claude (supervisor). Decomposes the intent into subagent instructions, reviews returned work, issues corrections. Never writes the build code.
- Layer 2, Codex subagents. Implement. Each subagent works only from the instruction it was given.
- Layer 3, the code. The final merged implementation.

## 3. The trace

Every signal between layers is logged as one JSON line in `trace/run-01.jsonl`.

Required fields per line.

- `ts` ISO 8601 timestamp
- `seq` monotonically increasing integer
- `from` and `to` one of `asish`, `claude`, `codex-<n>`, `repo`
- `type` one of `intent`, `decomposition`, `instruction`, `question`, `answer`, `progress`, `artifact`, `review`, `correction`, `resubmission`, `acceptance`, `handoff`
- `summary` one sentence, human readable
- `payload_ref` path to the full text of the message, stored under `trace/payloads/<seq>.md`

Rules.

- The trace is append only. Nothing is edited after the fact.
- Full message texts are preserved verbatim in payloads. The paper may quote them.
- Codex side events come from `codex exec --json` output, translated into this schema by the logger with the original preserved under `trace/raw/`.

## 4. Measuring drift, the snapshot method

Diffing prose against code directly is mushy. Instead, at each layer we capture a **task snapshot**, the same fixed questionnaire answered from that layer's viewpoint only, using only the information that layer received.

The questionnaire (identical at every layer).

1. What is being built, in one sentence?
2. List every required behavior.
3. List every explicit constraint (style, tools, limits).
4. List the edge cases that must be handled.
5. What is explicitly out of scope?
6. How will success be checked?

Snapshots are taken at four points.

- S0 from Asish's intent message (answered by a neutral agent reading only that message).
- S1 from Claude's decomposition (neutral agent reading only the instructions issued to subagents).
- S2 from each Codex subagent, asked to answer the questionnaire before writing code, seeing only its own instruction.
- S3 from the final code (a neutral agent reading only the merged code and tests, reverse engineering the answers).

Drift analysis. Each snapshot is compared to S0 item by item, and every requirement is classified as preserved, mutated (present but changed), dropped, or invented (present downstream but never asked for). The neutral agent doing snapshots and classification must be a model that took no part in the build, and its prompts are stored in `viz/prompts/` so the method is reproducible.

Recovery analysis. Every `correction` signal in the trace is linked to the requirement it restored, so the paper can show which drifted items the feedback loop caught and which reached the final code uncaught.

## 5. The figures

All figures are generated by scripts in `viz/` reading only the trace and snapshots. No hand drawn data.

1. **The lineage flow.** A Sankey style flow from S0 requirements through S1, S2, S3, each requirement a strand, colored by fate (preserved, mutated, dropped, invented). The signature figure.
2. **The connectome.** Nodes for Asish, Claude, each Codex agent, and the repo, edges weighted by message count, animated signal pulses on the live page.
3. **The firing timeline.** Raster plot, one row per participant, one tick per signal, correction signals highlighted, so the feedback loop reads as visible bursts.
4. **The drift table.** Per requirement, its wording at each layer side by side, with mutations highlighted. The most quotable artifact for PM readers.

Presentation. Animated SVG plus CSS on a GitHub Pages site under `site/`, static SVG exports embedded in the README, Mermaid sequence diagram for one correction loop.

## 6. Honesty rules

- One run, one case study. The paper claims nothing beyond what this trace shows.
- If drift turns out to be small, that is the finding and it is published as such.
- Related work cited, the 2026 MDPI orchestration survey, the orchestration traces paper (arXiv 2605.02801), GoAgent, BIGMAS, and the Nature Communications brain inspired planner.
- No AI attribution games, the repo says plainly how the study was run and who did what.

## 7. Build order

1. Trace logger (wraps the orchestration session, writes JSONL and payloads).
2. Dry run on a toy task to validate the schema end to end.
3. The real run, Asish provides the intent message cold.
4. Snapshots and drift classification.
5. Viz scripts, then figures, then the site, then the writeup.

## 8. Open items

- Asish must write the Layer 0 intent message himself, unprompted and uncoached, when we reach step 3.
- Choice of neutral snapshot model, decided at step 4 (must not be the supervisor or the workers).
